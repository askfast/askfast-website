VerifyMe

VerifyMe is used for verifying E-mail, Mobile and Landline phone of a user or a system. It has two processes (Initiate and Verify) to understand if the number is which a REST API imlemented for these two processes.

Initiate

This REST call initiates a dialog to push (by text or audio) the random verification code to the phonenumber specified 

POST <hostname>/products/verifyme/initiate
{
  "verificationMedium": "SMS | Call | Email | Auto",
  "verificationInfo": {
    "address": <address>,
    "message":" <message>",
    "subject":" <subject>",
    "senderName":" <sender name>"
   },
  "sessionTimeoutInSecs": <timeout>,
  "verificationCodeLength": <codeLength>
}
verificationMedium
medium used to send/play the verification code. Auto is an option which automatically predicts the medium based on the address field:
address
medium used to send/play the verification code.
Mobile number, an SMS triggered.
Fixedline number, a automated phone call is triggered.
For anything else, an Email is triggered.
message
This is parsed for a maching “[code]”. If found [code] is replaced by the random code that is generated.
If not, the message is appended with “Code: ”. For a Phone call, this audio URL is played before playing out the random code.
sessionTimeoutInSecs
This is the maximum time until which a verify API call is valid. Default value is 180s, incase this field is left empty/null.
verificationCodeLength
This is the length of the random number that has to be generated. Default value is 4, incase this field is left empty/null. 
POST <hostname>/products/verifyme/initiate
{
  "verificationMedium": "SMS",
  "verificationInfo": {
    "address": <address>,
    "message":"Following is your verification code: [code]",
    "subject":"oneline verification",
    "senderName":"online"
   }
}

Response

The following response is generated on success. The initiateResponse field indicates if there was an error in initiating the verification. If it is successful, a verificationInfo is also sent in the same response, which gives some extra details about the address being verified. 

{
  "initiateResponse": {
    "jsonrpc":"2.0",
    "id":"null",
    "result": {
      "0614765863": "SMS|0636465236|0614765863"
    }
   },
  "verficationInfo": {
    "verificationMedium": "SMS",
    "verificationStartTimestamp": 1384508760434,
    "address": "0614765863",
    "addressType": "MOBILE",
    "phoneNumberOrign": "Netherlands",
    "adapterConfigId": "786a8090-13b9-11e3-a435-00007£000001",
    "verified": false,
    "id": "8ac66391-38d3-4076-a27fbd4bd1498",
    "ownerKey": "apptestoneline"
  }
}
verificationStartTimestamp
timestamp in millis when this verification step was initiated.
adapterConfigId
corresponding adapter used to perform the step. This is useful when multiple adapters can be used to perform the step. E.g. multiple XMPP adapters.
verified
if the address is verified already or not.
id
a unique Id generated to mark this verification step. This is used in the Verify step explained below.

Verify

This API validates the random verification code provided by the user. Note that, to be validated successfully, this API has to be triggered before the sessionTimeoutInSecs field defined in the initiate step above. 

POST <hostname>/products/verifyme/verify
{
  "id": "<identifies this verification step>",
  "verificationCode": "<verification code send in the initiation step>"
}

Broadcast API:

This was created to add more flexibility to the old broadcast mechanism by supplying channel (E.g. SMS, EMAIL etc) specific adapter Ids. This is useful when a user has multiple adapters per channel and needs an option to pick one specific adapter.

Create a new broadcast:

The new implementation of Broadcast can be currently be done in the following ways:

  • Collection of numbers
  • Collection of phonebook references
    • ContactInfo references
    • Contact references
  • Collection of groups
  • CSV stream
  • CSV file upload
  • Excel file upload

Broadcast to a Collection of numbers, phonebook references or groups:

They are similar to each other with subtle differences. For collection of numbers, just use the “address” field. Similarly use the “contactInfoKey” field for refencing a phonebook contactInfo and “groupKey” field for referencing any groups. 

POST <hostname>/products/broadcastnew
{
  “addresses”: {
      “<AdapterType>”: [
          {“address”:“<email address1>”}
          {“address”:“<email address2>”}
       ]
   },
   “adapterList”: {
       "<AdapterType>”:“<calling adapterId>”
   },
   “senderName”:“<senderName (used for all but CALL channel)>”,
   “message”:“Test message”,
   “audio”:“http://autioURL.wav”,
   “retryMethod”:“AUTO|MANUAL”,
   “retryLimit”:<retryLimit>,
   “broadcastName”:“<Custom name to describe this broadcast>”,
   “emailSubject”:“<email subject (used only for EMAIL channel)>”
}

where

AdapterType:
[SMS | CALL | EMAIL | XMPP | FACEBOOK | TWITTER]
Mandatory
field which explicilty describes which channel is used to broadcast the message.
addresses
Mandatory field which describes all the address to which the message must be broadcasted. At Least one of the fields: address, contactInfoKey or groupKey must exist per channel. Only one broadcast message is sent however if the same address is found in multiple fields, then broadcast is done to only one.
adapterList
Optional field which gives an option to explicitly specifies which adapter to use for a particular channel. If not adapter is specified a default adapter that belongs to the currently logged in user is picked.
retryMethod (Manual)
[Auto|Manual] is used to indicate the method used when any broadcast send action (not delivery!) fails.
retryLimit(5)
is used to limit the number of auto-retries. This is not applicable if Manual RetryMethod is used.
message(“null”)
Simple text (or URL to a question (which can be used for allowing users to reply on the message)) message that has to be broadcasted.
audio(“null”)
(Specific to Voice only) Simple audio URL to a wav file (URL should end with “wav”) used to supercede the message text when performing a broadcast to text as well as voice mediums (SMS and CALL) in one call. If this is found to be null or not ending with “wav”, then the voice channel would speak out the message text.
broadcastName(“null”)
Custom name to describe this broadcast.
emailSubject(“null”)
Email subject (used only for EMAIL channel.
senderName(“null”)
Specific senderName used with all mediums except Call. Please be aware that for an SMS, defining a senderName will obviously remove the possibility to reply to a message.

Optional query parameters:

isPersisted(true)
[true|false] Flag to confirm if this broadcasting has to be persisted. If its not persisted options to resent/retry the broadcast would not be possible.

Broadcast retry failed:

This will retry all the failed broadcast that has occured in a previous attempt.

POST <hostname>/products/broadcastnew/retryfailed/<broadcastId>

where,

broadcastId
is the id associated with any previous broadcast, received in the response of a broadcast call.

Re-Broadcast:

This will repeat a prior stored broadcast.

POST <hostname>/products/broadcastnew/resent/<broadcastId>

where,

broadcastId
is the id associated with any previous broadcast, received in the response of a broadcast call.

Fetch all the broadcasts:

Fetches a collection of all previously broadcasted messages.

GET <hostname>/products/broadcastnew

Fetch a specific broadcast:

Fetches a specific broadcast based on its id

GET <hostname>/products/broadcastnew/<broadcastId>

Delete a specific broadcast:

Deletes a specific broadcast based on its id

DELETE <hostname>/products/broadcastnew/<broadcastId>

Prefix search for a contactInfo, firstName or lastName

This API can be used to fetch all the broadcasts with a specific query string as the prefix of broadcast name. 

GET <hostname>/products/broadcastnew/prefixsearch/{query}
Response:
[
   {
       "id":"<broadcast id>",
       "match":"<fieldName>:// <fieldValue>"
   }
]

Update a specific broadcast:

Updates a specific broadcast based on its id, so that it can be used to send a broadcast to a new set of addresses. This will overwrite the previously created broadcast entity. 

PUT <hostname>/products/broadcastnew/<broadcastId>
{
   broadcast_playload
}

File upload:

This interface, is used to upload a file into the blobstore of the app engine. Of course, the user has to be registered and logged in. We can create a username and password for testing. 

GET <hostname>/blobstore/<someRandomFileName>?!

This generates an HTML page with option to select a file from the local computer. A POST request is then performed on a google blobstore upload servlet generated path: 

POST <hostname>/_ah/upload/<blobstoreUploadKey>

Example:

 <hostname>/_ah/upload/ag1hc2tmYXN0bWFya2V0chwLEhVfX0Jsb2JVcGxvYWRTZXNzaW9

Initiate a call

To initiate an outbound call, perform the following request. This will ensure that your account and the Question (accessed via the URL) is setup properly. The blue fields are mandatory. 

POST http://ask-charlotte.appspot.com/rpc
{
   "id":1,
   "method":"outboundCall",
   "params": {
       "publicKey":"<publicKey>",
       "privateKey":"<privateKey>",
       "adapterID":"<adapterID>",
       "address":"<address>",
       "url":"<URL to fetch the Question from>",
       "senderName":"<sender Name>"
   }
}

where,

publicKey, privateKey and adapterID
IDs linked to the registered account.
address
Address to which an outboundCall is to be performed. E.g. phone-number in case of SMS or PhoneCall, email address in case of Email or XMPP etc.
url
The URL to fetch the Question from. The ASK-Fast Server performs a GET request on this URL and expects a response in the Question format.

To initiate outBound broadcast calls use the following API:

POST http://ask-charlotte.appspot.com/rpc
{
   "method":"outboundCallWithMap",
   "params": {
       "adapterID":"<adapterId>",
       "addressMap": {
           "<address1>":"Alice"
       },
       "url":"<URL to fetch the Question from>",
       "publicKey":"<publicKey>",
       "privateKey":"<privateKey>",
       "senderName":"<sender Name>"
   }
}

Advanced: Trigger events

The following event callbacks can be used to capture any events in the communication channel. 

"event_callbacks": [
 {
   "event":"[answered|timeout|exception|hangup]",
   "callback":"<url>"
 }
]

where,

answered
used to trigger a callback when a phone call is answered by the callee.
timeout
when a phone call is missed by the callee.
exception
when a phone call is not triggered for miscellaneous reasons.
hangup
when the callee hangs up a phone call.

The POST request initiated on the callback url carries the following payload:

{
 "responder":"<id url>",
 "question_id":<originating question>,
 "event":"<originating event>",
 "extras": {
     <optional/additional information about this event>
 },
 "message":"<string describing the event, error message>"
}

where,

extras, has a key, value pair of any additional information related to this event. E.g.

"extras": {
 "startTime":1382623574000,
 "releaseTime":1382624584000,
 "answerTime":1382623584000
}

carries the startTime (timestamp at which the phone call started), answerTime (timestamp at which the phone call was answered) and releaseTime (timestamp at which the phone call was hang up).
